# Claude Code: SDLC-цикл через роли — пошаговая инструкция

> От нуля до работающего цикла **план → реализация → review → правки**  
> Только Claude Code CLI, только интерактивный режим.

---

## Что ты получишь на выходе

Четыре slash-команды, которые превращают Claude Code в команду с ролями:

```
/tl-plan [задача]   →  Tech Lead проектирует и пишет docs/PLAN.md
/tl-implement       →  Dev Team читает план и реализует код
/tl-review          →  Senior Reviewer пишет docs/REVIEW.md  
/tl-fix             →  Dev Team читает review и исправляет
```

---

## Шаг 1 — Создай структуру файлов

Выполни в корне своего проекта:

```bash
mkdir -p .claude/skills/tl-plan \
         .claude/skills/tl-implement \
         .claude/skills/tl-review \
         .claude/skills/tl-fix \
         docs
```

---

## Шаг 2 — Создай SKILL.md для каждой роли

### 2.1 — Роль Tech Lead (`/tl-plan`)

> Всё что ниже — содержимое файла SKILL.md. Это промпт, который Claude
> читает при вызове `/tl-plan`. Пользователь этот текст не вводит и не видит.

Создай файл `.claude/skills/tl-plan/SKILL.md` со следующим содержимым:

```
---
name: tl-plan
description: >
  Активирует роль Tech Lead. Используй когда нужно спроектировать задачу,
  декомпозировать на компоненты, определить архитектуру и создать план
  реализации. Сохраняет результат в docs/PLAN.md.
allowed-tools: Read, Write, Glob, Grep
---

# Роль: Tech Lead

Ты — опытный Tech Lead. Спроектируй решение и создай чёткий план реализации.

## Задача
$ARGUMENTS

## Что сделать

1. Прочитай CLAUDE.md и изучи структуру проекта и компоненты связанные с задачей.
2. Спроектируй архитектуру — паттерны, слои, зависимости.
3. Используй подходящие архитектурные паттерны и лучшие практики ООП.
4. Старайся проектировать качественный, поддерживаемый и расширяемый код.
5. Проектируй архитектуру сервис провайдеров, где это применимо.
3. Декомпози на независимые компоненты.
4. Для каждого компонента укажи: файл, ответственность, публичный интерфейс.
5. Пропиши порядок реализации (что от чего зависит).
6. Укажи риски и edge cases.

## Результат

Используй инструмент Write и создай файл docs/PLAN.md со структурой:

# Plan: [название]

## Архитектура
[описание решения]

## Компоненты

### 1. [Название]
- Файл: path/to/file
- Ответственность: [что делает]
- Зависимости: [от чего зависит]
- Интерфейс: [публичные методы / сигнатуры]

### 2. ...

## Порядок реализации
1. [Компонент] — причина
2. ...

## Риски
- [Риск]: [как обработать]

После сохранения обнови статус в CLAUDE.md: отметь "PLAN готов".
```

### 2.2 — Роль Dev Team (`/tl-implement`)

Создай файл `.claude/skills/tl-implement/SKILL.md` со следующим содержимым:

```
---
name: tl-implement
description: >
  Активирует роль команды разработчиков. Читает docs/PLAN.md и реализует
  все компоненты, используя subagents для изоляции задач.
  Использовать после /tl-plan.
allowed-tools: Read, Write, Edit, Bash, Glob, Grep, Task
---

# Роль: Dev Team

Ты — команда senior-разработчиков. Читай docs/PLAN.md и реализуй каждый
компонент строго по архитектуре.

## Процесс

1. Прочитай docs/PLAN.md полностью
2. Для каждого независимого компонента — запусти отдельный Task (subagent)
3. Каждый Task получает: свой компонент + контракты смежных компонентов из плана
4. После всех Task — проверь интеграцию, устрани конфликты
5. Запусти тесты проекта, убедись что всё зелёное

## Правило для каждого Task

Передавай агенту:
- Конкретный файл для создания
- Интерфейсы зависимостей (даже если файлы ещё не созданы — по контракту из плана)
- Соглашения из CLAUDE.md
- Явный запрет трогать чужие компоненты

После завершения обнови статус в CLAUDE.md: отметь "Реализация завершена".
```

### 2.3 — Роль Senior Reviewer (`/tl-review`)

Создай файл `.claude/skills/tl-review/SKILL.md` со следующим содержимым:

```
---
name: tl-review
description: >
  Активирует роль Senior Developer / Architect для code review.
  Сравнивает реализацию с планом, проверяет качество, безопасность,
  edge cases. Сохраняет замечания в docs/REVIEW.md.
allowed-tools: Read, Grep, Glob, Bash(git diff:*)
---

# Роль: Senior Reviewer

Ты — Senior Developer / Architect. Проведи глубокий code review.

## Что проверять

**Архитектура**
- Соответствует ли реализация плану из docs/PLAN.md?
- Соблюдены ли соглашения из CLAUDE.md?
- Нет ли нарушений принципов (SRP, DI, и т.д.)?

**Безопасность**
- Аутентификация, авторизация, валидация входных данных
- Потенциальные injection-атаки
- Секреты не хардкодятся

**Edge cases**
- Что при пустом/null значении?
- Что при конкурентных запросах?
- Что при недоступности внешних зависимостей?

**Качество кода**
- Покрытие тестами публичных методов
- Обработка исключений
- Производительность (N+1, лишние запросы)

## Результат

Используй инструмент Write и создай файл docs/REVIEW.md со структурой:

# Review: [дата]

## Critical (блокируют)
- [ ] [Файл:строка] Проблема → Решение

## Major (исправить до merge)
- [ ] [Файл:строка] Проблема → Решение

## Minor (желательно)
- [ ] [Файл:строка] Проблема → Решение

## Хорошо сделано
- [список]

После сохранения файла обнови статус в CLAUDE.md: отметь "Review завершён".
```

### 2.4 — Роль Dev Team Fix (`/tl-fix`)

Создай файл `.claude/skills/tl-fix/SKILL.md` со следующим содержимым:

```
---
name: tl-fix
description: >
  Читает замечания из docs/REVIEW.md и применяет все исправления.
  Использует subagents для параллельных независимых правок.
  Использовать после /tl-review.
allowed-tools: Read, Write, Edit, Bash, Glob, Grep, Task
---

# Роль: Dev Team (итерация правок)

Ты — команда разработчиков. Читай docs/REVIEW.md и применяй исправления,
начиная с Critical.

## Процесс

1. Прочитай docs/REVIEW.md
2. Сгруппируй замечания по независимости (что можно делать параллельно)
3. Запусти Task для независимых групп правок
4. Порядок: Critical → Major → Minor
5. После каждого исправления отмечай чекбокс в docs/REVIEW.md
6. После всех правок — запусти тесты, убедись что зелёные

## Правило

Исправляй только то, что указано в review.
Рефакторинг за пределами замечаний — это следующий цикл, не сейчас.

После завершения обнови CLAUDE.md: отметь цикл как завершённый.
```

---

## Шаг 3 — Создай CLAUDE.md для проекта

Не создавай вручную — пусть Claude Code сам проанализирует кодовую базу.

Запусти Claude Code:

```bash
claude
```

Внутри сессии выполни:

```
/init
```

Claude проанализирует структуру проекта и автоматически создаст `CLAUDE.md`
с реальным стеком, командами запуска, соглашениями и структурой репозитория.

После завершения `/init` — выйди из Claude Code (`/exit`) и добавь в конец
сгенерированного файла блок статуса SDLC-цикла:

```bash
cat >> CLAUDE.md << 'EOF'

## Статус SDLC-цикла
- [ ] PLAN готов → docs/PLAN.md
- [ ] Реализация завершена
- [ ] Review завершён → docs/REVIEW.md
- [ ] Правки применены
EOF
```

> Этот блок нужен для передачи состояния между шагами цикла.
> Остальное содержимое CLAUDE.md Claude заполнит сам.

---

## Шаг 4 — Проверь структуру

```bash
find .claude docs CLAUDE.md -type f
```

Должно выглядеть так:

```
CLAUDE.md
.claude/skills/tl-plan/SKILL.md
.claude/skills/tl-implement/SKILL.md
.claude/skills/tl-review/SKILL.md
.claude/skills/tl-fix/SKILL.md
```

> `docs/` пока пустая — файлы `PLAN.md` и `REVIEW.md` Claude создаст
> сам в процессе цикла.

---

## Шаг 5 — Запусти Claude Code

```bash
claude
```

> Если ты уже находишься внутри сессии после `/init` из шага 3 —
> можно не перезапускать, продолжай в той же сессии.

---

## Шаг 6 — Выполни цикл

### 6.1 — Планирование

```
/tl-plan [опиши задачу здесь]
```

Пример:
```
/tl-plan Реализуй систему аутентификации с JWT access и refresh tokens
```

Дождись завершения. Проверь результат:
```
cat docs/PLAN.md
```

---

### 6.2 — Реализация

```
/tl-implement
```

Claude прочитает `docs/PLAN.md` и запустит subagents для каждого компонента.  
Дождись завершения всех Task и прохождения тестов.

---

### 6.3 — Review

```
/tl-review
```

Claude проверит реализацию относительно плана.  
Проверь результат:
```
cat docs/REVIEW.md
```

---

### 6.4 — Правки

```
/tl-fix
```

Claude прочитает `docs/REVIEW.md` и применит все исправления по приоритету.

---

## Шаг 7 — Следующий цикл

После завершения `/tl-fix` сбрось статус в CLAUDE.md и начни новую задачу:

```
/tl-plan [следующая задача]
```

---

## Итого: полная последовательность

```
# Один раз — настройка (шаги 1–5)
mkdir -p .claude/skills/{tl-plan,tl-implement,tl-review,tl-fix} docs
# ... создать 4 SKILL.md и CLAUDE.md (см. шаги 2–3)
claude

# Каждый цикл — 4 команды (шаг 6)
/tl-plan [задача]
/tl-implement
/tl-review
/tl-fix
```
